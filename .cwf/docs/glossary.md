## CWF

**Abbrev**: CWF = Coding with Files
The name of this project and documentation system. Pronounced as three letters (C-W-F),
not as a word. Formerly called CIG (Code Implementation Guide) — CIG references in older
files are equivalent.

---

## cwf- prefix

A namespace convention applied to all CWF-owned artefacts in `.claude/` to prevent
clashes with other skills, plugins, or rules in the same project. CWF skills use the
prefix `cwf-` in their directory name (e.g. `.claude/skills/cwf-task-plan/`), and CWF
rules use it in their filename (e.g. `.claude/rules/cwf-workflow-files.md`). Third-party
skills and rules should not use the `cwf-` prefix.

---

## checkpoint commit

A git commit made at the end of each workflow phase, staging only that phase's workflow
file. Checkpoint commits preserve incremental progress on the checkpoints branch and are
later squashed into a single task commit at retrospective time.
**See**: `.cwf/docs/skills/checkpoint-commit.md`, [checkpoints branch](#checkpoints-branch)

---

## checkpoints branch

**Not**: "checkpoint branch" (missing the s)
A git branch named `<type>/<num>-<slug>-checkpoints` that preserves the full sequence of
checkpoint commits for a task. It is created at retrospective time from the task branch
history and is never merged to main — it exists for reference and audit only.
**See**: [task branch](#task-branch), [checkpoint commit](#checkpoint-commit)

---

## rule

A Claude Code path-scoped instruction file in `.claude/rules/<name>.md`. Rules auto-load
into the agent's context when it operates on files matching the glob pattern in the rule's
YAML frontmatter. CWF rules use the `cwf-` prefix (e.g. `cwf-workflow-files.md`) to avoid
namespace clashes. Rules are advisory — they guide the agent but cannot enforce compliance.
**See**: [cwf- prefix](#cwf--prefix), [skill](#skill)

---

## skill

A Claude Code slash command defined as a markdown file in `.claude/skills/<name>/SKILL.md`.
Skills are the primary interface for invoking CWF workflow phases (e.g. `/cwf-task-plan`,
`/cwf-implementation-exec`). Each skill contains scope, workflow steps, and success criteria
for one phase.

---

## slug

The URL-safe fragment derived from a task description, used in directory names and branch
names. Generated by lowercasing, replacing spaces with hyphens, removing special characters,
and truncating to 50 characters.
Example: "Add user authentication" → `add-user-authentication`

---

## squash commit

The single consolidated git commit that replaces all checkpoint commits on the task branch
at retrospective time. Created with `git reset --soft <base> && git commit`. The squash
commit is what gets merged to main; the individual checkpoint commits are preserved only
on the checkpoints branch.
**See**: [checkpoints branch](#checkpoints-branch), [task branch](#task-branch)

---

## task branch

The primary git branch for a task, named `<type>/<num>-<slug>` (e.g.
`hotfix/87-create-cwf-terminology-glossary`). Work is done here throughout the task.
At retrospective time, all commits on the task branch are squashed into one before
merging to main.
**See**: [squash commit](#squash-commit), [checkpoints branch](#checkpoints-branch)

---

## wf

**Abbrev**: wf = workflow
Short form of "workflow", used in filenames (e.g. `wf step files`) and inline prose.
`wf step files` refers to the lettered workflow files in a task directory
(a-task-plan.md, d-implementation-plan.md, etc.).
