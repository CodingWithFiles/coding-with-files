## Goal
Remove the vestigial top-level `version` field from `cwf-project.json` (and its shipped template) so `.cwf/version` is the single, unambiguous record of the installed CWF version — resolving the version-drift backlog item by deletion rather than by a drift check.

## Background
Backlog item "Resolve cwf-project.json version drift vs .cwf/version" (resolution path A). An external user found `.cwf/version` bumped to `v1.0.114` while `cwf-project.json` still recorded `version: v1.0.95`. Investigation (carried over from the discarded path-B planning) established that the top-level `version` field is **vestigial**:
- **Zero readers** anywhere in `.cwf/`, `t/`, or `.claude/` (no `$config->{version}` access).
- **Not schema-required** — `CWF::Validate::Config` neither requires nor rejects it.
- **Hardcoded-stale by construction** — `.cwf/templates/cwf-project.json.template` ships a literal `"version": "v0.2.1"`; fresh installs get `v0.2.1` regardless of the real installed version, so the field never tracked anything in current code.

A drift check (path B) would therefore nag users about a decorative field with no meaningful remediation. Removing it eliminates the second version claimant entirely — drift becomes impossible.

## Risk Assessment
### High Priority Risks
- *(none)* — the field has no readers, so removal cannot change behaviour.

### Medium Priority Risks
- **Risk 1**: A hidden consumer exists that the grep missed (e.g. an external/user script, or a consumer that reads the raw JSON outside this repo).
  - **Mitigation**: Scope the claim to this repo (the only thing we control). The field is decorative here; document that any out-of-repo consumer was reading a hardcoded `v0.2.1` and so was already not getting useful data. Re-run the reader grep at implementation-plan time to confirm nothing landed since.
- **Risk 2**: The field is laid down/regenerated by install tooling, so removing it from the template alone leaves it reappearing.
  - **Mitigation**: Confirmed the template is the laydown source and is not regenerated with a `version` key elsewhere; re-verify `install.bash` / `cwf-apply-artefacts` / `cwf-init` at implementation-plan time.
